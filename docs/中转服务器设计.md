# 中转服务器设计

## 定位

服务器模式下的云中转节点。小车和前端都连它，数据经它转发。同时做简单存储。

## 架构

```
小车                          中转服务器                      前端
────                          ──────────                      ────

[遥测采集] ──WS──→  /ws/car/telemetry  ──转发──→ /ws/client/telemetry ──→ [显示]
[帧数据]   ──WS──→  /ws/car/frame      ──转发──→ /ws/client/frame     ──→ [显示]
[GPS]      ──WS──→  /ws/car/gps        ──转发──→ /ws/client/gps       ──→ [地图]

[执行器]   ←──WS──  /ws/car/control    ←──转发── /ws/client/control   ←── [操控面板]

                    ├── 保存遥测/帧/GPS → sessions/
                    └── 后端 API (重建/检测) → 同机或转发
```

小车和前端的连接都是 WebSocket，服务器在中间做消息路由。

## 数据流

### 上行（小车 → 前端）

```
小车 WS 发 JSON
  → 服务器收到，做三件事：
    1. 转发给所有已连接的前端客户端
    2. 写入 session 文件
    3. （可选）转发给后端 API 做重建/检测
```

### 下行（前端 → 小车）

```
前端 WS 发控制指令 JSON
  → 服务器收到，转发给指定小车
```

## WebSocket 端点

| 端点 | 方向 | 说明 |
|------|------|------|
| `/ws/car` | 小车 → 服务器 | 小车统一入口，消息带 type 字段区分遥测/帧/GPS |
| `/ws/client` | 前端 ↔ 服务器 | 前端统一入口，收发所有数据 |

只用两个 WS 端点，消息内部用 `type` 字段区分，简化连接管理。

## 消息格式

### 小车 → 服务器

```jsonc
// 遥测 (10Hz)
{ "type": "telemetry", "timestamp": 1.0, "speed": 2.0, "steering_angle": 15.0 }

// 帧数据 (≤1fps)
{ "type": "frame", "timestamp": 1.5, "image": "<base64>", "depth_map": "<base64>" }

// GPS (1Hz)
{ "type": "gps", "timestamp": 2.0, "lat": 39.9042, "lng": 116.4074, "alt": 50.0 }
```

### 服务器 → 前端（转发同上，加 car_id）

```jsonc
{ "type": "telemetry", "car_id": "car-001", "timestamp": 1.0, "speed": 2.0, "steering_angle": 15.0 }
```

### 前端 → 服务器 → 小车（控制指令）

```jsonc
// 操控小车
{ "type": "control", "car_id": "car-001", "command": "set_speed", "value": 1.5 }
{ "type": "control", "car_id": "car-001", "command": "set_steering", "value": 10.0 }
{ "type": "control", "car_id": "car-001", "command": "stop" }

// 查询状态
{ "type": "query", "car_id": "car-001", "field": "battery" }
```

### 小车 → 服务器（响应）

```jsonc
{ "type": "response", "car_id": "car-001", "field": "battery", "value": 85 }
```

## GPS 数据

| 字段 | 类型 | 说明 |
|------|------|------|
| timestamp | float | Unix 时间戳 |
| lat | float | 纬度 |
| lng | float | 经度 |
| alt | float | 海拔 (m)，可选 |

频率 1Hz。前端界面一的地图区域显示小车轨迹。

## 数据保存

服务器收到每条消息都写入本地 session 文件。格式同 `docs/数据保存与回放格式.md`：

```
sessions/
├── car-001/
│   ├── 20240718_120000/
│   │   ├── manifest.json       # 遥测时间线 + GPS + 帧索引
│   │   ├── frames/
│   │   │   ├── 00001.jpg
│   │   │   ├── 00001.png
│   │   │   └── ...
│   │   └── gps_track.json      # GPS 轨迹 [{"ts":1.0,"lat":...,"lng":...}, ...]
```

支持按 car_id 和日期组织，前端回放时可选择任意 session。

## 与后端 API 的关系

中转服务器和后端 API（Python FastAPI）可以：

**方案 A — 同机部署（推荐开发期）**

中转服务器收到帧数据后，内部 HTTP 调用 `localhost:8000/api/vehicle/frame` 触发重建和检测。前端仍然通过中转服务器获取重建结果，或直接调后端 API。

**方案 B — 独立部署**

中转服务器只做转发 + 存储。后端 API 单独运行，前端直连后端。GPS 和操控指令与后端无关。

## 技术选型

Python + FastAPI + WebSocket，和现有后端统一技术栈。

```
server-relay/
├── main.py              # FastAPI + WS 入口
├── car_manager.py       # 小车连接管理 (car_id → WS 映射)
├── client_manager.py    # 前端连接管理 (广播)
├── session_writer.py    # 数据保存（复用 data-saver 格式）
├── config.yaml          # 服务器配置
└── requirements.txt     # fastapi + uvicorn + pyyaml
```

## 配置

```yaml
# server-relay/config.yaml

server:
  host: "0.0.0.0"
  port: 9000              # 中转端口（与后端 8000 分开）

save:
  enabled: true            # 是否保存数据
  path: "./sessions"       # 保存路径
  max_per_car: 100         # 每车最多保留 session 数

backend:                   # 后端 API 地址（方案 A 时使用）
  host: "127.0.0.1"
  port: 8000
```

## 实施优先级

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P0 | 小车 ↔ 前端消息转发 | 核心功能，没有它服务器模式不工作 |
| P0 | 数据保存 | 收到就存，不需要额外逻辑 |
| P1 | GPS 收发 | 独立消息类型，加一个 type 即可 |
| P2 | 控制指令转发 | 前端 → 小车，需要小车端配合 |
| P2 | 多车支持 (car_id) | 一台服务器服务多辆小车 |
