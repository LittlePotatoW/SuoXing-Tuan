## 一、Bug 和严重错误

### 1.1 后端模块 (backend/)

**B1. fusion/datafusion.py:36 - 文档字符串错误**
- 位置：`fusion/datafusion.py:36`
- 问题：参数注释写的是 `[x,y,z, qw,qx,qy,z]`，应为 `[x,y,z, qw,qx,qy,qz]`
- 影响：代码维护者可能误解参数格式

**B2. reconstruction/engine.py:195 - None 解引用风险**
- 位置：`reconstruction/engine.py:193-195`
- 问题：`_reconstruct_surface()` 返回 `None` 时，`self._latest_mesh` 仍为之前的值，逻辑不一致
- 影响：统计数据可能不准确

**B3. realtime/fusion_router.py:169 - 异常捕获过于宽泛**
- 位置：`realtime/fusion_router.py:169`
- 问题：`except: continue` 捕获所有异常（包括 SystemExit、KeyboardInterrupt）
- 影响：可能掩盖严重错误，导致问题难以排查
- **建议**：改为 `except Exception: continue`

**B4. state_estimation/estimator.py:259 - 零速检测逻辑缺陷**
- 位置：`state_estimation/estimator.py:259`
- 问题：`_last_valid_vel` 在检测之前就被更新，导致零速检测无法生效
- 影响：状态估计精度下降
- **建议**：在更新 `_last_valid_vel` 之前进行检测

**B5. reconstruction/projector.py - 畸变系数缺失**
- 位置：`reconstruction/projector.py`
- 问题：`project_pixel_defect()` 和 `_pixel_to_camera_3d()` 完全没有考虑镜头畸变
- 影响：缺陷坐标反算精度下降

**B6. reconstruction/routes.py - WebSocket 客户端管理线程不安全**
- 位置：`reconstruction/routes.py:217-227`
- 问题：`_ws_clients.append(ws)` 和 `_ws_clients.remove(ws)` 无锁保护
- 影响：并发访问时可能导致竞态条件

**B7. realtime/fusion_router.py - _run_yolo 设计问题**
- 位置：`realtime/fusion_router.py:156-185`
- 问题：`DefectProjector` 在每次推理时重新创建，且使用默认内参
- 影响：缺陷投影精度低

## 二、可改进的设计/架构

### 2.1 后端模块

**D1. 单例模式滥用**
- 问题：`realtime/fusion_router.py`、`reconstruction/routes.py`、`state_estimation/router.py` 均使用全局单例
- 影响：无法进行单元测试、模块间隐式耦合、无法支持多场景并行处理
- 建议：使用依赖注入模式，通过 FastAPI 的 `Depends()` 或第三方库（如 `injector`）管理依赖

**D2. 坐标变换代码重复**
- 问题：四元数→旋转矩阵、旋转矩阵→四元数、位姿→变换矩阵等函数在 `common/transform.py`、`fusion/datafusion.py`、`reconstruction/projector.py` 中重复实现
- 影响：代码冗余，维护成本高，不同实现可能存在细微差异
- 建议：将所有坐标变换统一到 `common/transform.py`

**D3. 同步/异步混合架构混乱**
- 问题：`df.process()`、`eng.add_frame()`、`eng.get_result()` 都使用 `threading.RLock`，在 `run_in_executor` 中连续调用多个持锁操作
- 影响：可能导致死锁或锁竞争，没有利用 asyncio 的并发优势

**D4. 配置管理分散**
- 问题：可调参数散落在多个文件中（`VOXEL_SIZE`、`MAX_ACCEL`、`HOST`、`PORT` 等）
- 建议：集中到 `config.py` 或 `.env` 文件，使用 `pydantic-settings` 管理

**D5. 缺少接口定义**
- 问题：`DataFusionManager` 的 `__init__` 依赖 `StateEstimator`，但没有定义抽象接口
- 建议：定义抽象基类（ABC），降低耦合度

**D11. 类型定义不完整**

## 四、性能优化点

### 4.1 后端模块

**P1. common/projection.py - 深度图构建效率低**
- 位置：`common/projection.py:105-109`
- 问题：使用 Python 循环遍历每个像素构建深度图，时间复杂度 O(n)
- 优化方案：使用 NumPy 向量化操作

**P2. fusion/coloring.py - 颜色采样循环慢**
- 位置：`fusion/coloring.py:191-198`
- 问题：Python 循环逐个更新颜色，效率低
- 优化方案：先筛选需要更新的点，再批量处理

**P3. state_estimation/estimator.py - 位置查询效率低**
- 位置：`state_estimation/estimator.py:189-206`
- 问题：线性搜索查询位置，时间复杂度 O(n)
- 优化方案：使用二分查找（`bisect` 模块），时间复杂度降至 O(log n)

**P4. reconstruction/engine.py - 点云合并内存峰值**
- 位置：`reconstruction/engine.py:166`
- 问题：`np.vstack(blocks)` 创建临时大数组，导致内存峰值
- 优化方案：先计算总点数，预分配数组，再逐个拷贝

**P5. fusion/coloring.py - 最近邻搜索效率**
- 位置：`fusion/coloring.py:226-235`
- 问题：每个灰色点对所有有效点计算距离，时间复杂度 O(M*K)
- 优化方案：使用 `scipy.spatial.KDTree`，时间复杂度降至 O(M*log K)

---

### 4.2 前端模块

**P6. ReconstructionViewer.vue - 不必要的渲染循环**
- 问题：`loop` 函数每帧都调用 `renderer.render()`，即使场景没有变化
- 建议：添加 dirty flag，只在场景发生变化时才进行渲染

**P7. ReconstructionViewer.vue - 线框渲染效率低**
- 问题：为每个 mesh 创建两个几何体（实体和线框），双倍消耗 GPU 资源
- 建议：使用 `EdgesGeometry` 和 `LineSegments` 替代独立的线框 mesh

**P8. ReconstructionViewer.vue - 几何体复用**
- 问题：`addCracks` 函数为每个裂缝创建独立的 `SphereGeometry`
- 建议：使用 `InstancedMesh` 复用几何体和材质，大幅减少 draw call

**P9. ImageViewer.vue - 重复绘制优化**
- 问题：每次 `imageSrc` 或 `detections` 变化时都重新加载图片并绘制整个 Canvas
- 建议：将图片加载和检测框绘制分离

**P10. RealtimeView.vue - 轮询效率优化**
- 问题：`activePoll` 函数每次轮询都创建新的 HTTP 请求
- 建议：使用 `keep-alive` 连接，实现增量更新，或使用 WebSocket 替代轮询

---

### 4.3 模型训练模块

**P11. dataset.py - 单线程数据处理**
- 问题：`convert_mask_dataset` 使用单线程顺序处理图片
- 建议：使用 `concurrent.futures` 或 `multiprocessing` 进行并行处理

**P12. predict.py - 缺少批量推理优化**

**P13. train.ipynb - ONNX 导出缺少优化**
- 问题：ONNX 导出时没有启用 `dynamic` 批处理尺寸
- 建议：添加 `dynamic=True` 参数

### 4.4 测试和脚本模块

**P14. simulator.py - 图片生成性能**
- 问题：使用 Python 双重循环遍历每个像素生成图片

**P15. simulator.py - 点云生成性能**

**N8. 错误处理不统一**
- 问题：`App.vue`、`DetectionView.vue`、`ReconstructionView.vue`、`RealtimeView.vue` 的错误处理方式不一致
- 建议：实现统一的错误处理机制，如全局错误提示组件

**N9. 命名规范不严格**
- 问题：单字母变量名（`d`, `j`, `m`）、占位页面命名为 `Page2.vue`、变量命名混合中英文
- 建议：使用有意义的变量名，占位页面使用 `PlaceholderView.vue`

**N10. 资源清理不完整**
- 问题：多个组件在卸载时没有完全清理资源（图片对象、几何体、WebSocket 事件监听器）
- 建议：在 `onUnmounted` 中完善资源清理逻辑

**N11. 样式一致性缺失**
- 问题：按钮样式在不同组件中不一致
- 建议：创建统一的按钮组件或样式类

**N12. 路由配置包含占位页面**
- 问题：路由中包含 `page2`, `page3`, `page4` 等占位页面
- 建议：移除占位路由或标记为开发中状态

**N13. 环境变量使用不当**
- 问题：后端地址硬编码在 `apiClient.ts` 和 `relayConfig.ts` 中
- 建议：使用 `.env` 文件和 Vite 的环境变量机制
