## 一、严重 Bug 和错误

### 1.1 后端并发安全问题（Critical）

**位置**: `backend/main.py`, `backend/reconstruction/routes.py`

**问题**:
- `ws_clients` 列表在 `_broadcast` 函数中遍历时有使用锁，但在连接和断开时也需要使用锁保护
- `ReconstructionEngine` 的 `need_rebuild` 和 `rebuild_triggered` 计算在锁外，但使用了锁保护的共享变量
- `routes.py` 中 `_ws_clients.append(ws)` 没有使用锁保护

**影响**: 高并发场景下数据损坏、`RuntimeError: dictionary changed size during iteration`、服务崩溃

---

### 1.2 后端资源泄漏（Critical）

**位置**: `backend/reconstruction/engine.py`, `backend/reconstruction/routes.py`

**问题**:
- `_camera_trail` 列表只增不减，长时间运行后内存占用持续增长
- 在 `load_scene` 中直接重新赋值 `engine = ReconstructionEngine()`，丢失之前的状态且无法正确清理资源

**影响**: 内存泄漏、状态不一致

---

### 1.3 后端 CUDA 清理逻辑缺陷（High）

**位置**: `backend/inference.py:90-93`

**问题**: `_has_cuda()` 方法在内部导入 `torch`，如果 CUDA 不可用，后续的 `torch.cuda.empty_cache()` 调用可能会失败

**影响**: 模型卸载时可能抛出异常，导致资源泄漏

---

### 1.4 lidar_camera_fusion 切向畸变计算错误（Critical）

**位置**: `lidar_camera_fusion/lidar_camera_fusion.py:956-957`

**问题**: 投影函数中切向畸变的两个系数计算表达式完全相同，导致 p2 参数失效

```python
two_p1_xy = 2.0 * p1 * x_norm * y_norm
two_p2_xy = 2.0 * p2 * x_norm * y_norm  # ← 应该使用不同的公式
```

**影响**: 当相机存在切向畸变时，投影结果会明显偏移

---

### 1.5 lidar_camera_fusion 强度着色模式失效（Critical）

**位置**: `lidar_camera_fusion/lidar_camera_fusion.py:1107-1109`

**问题**: `color_map="intensity"` 模式回退到深度着色，而非使用点云强度数据

```python
elif color_mode == "intensity":
    colors = color_map_distance(depths)  # ← 错误：应该使用 intensity
```

**影响**: 用户选择强度着色时，实际看到的仍是距离伪彩色，功能完全失效

---

### 1.6 前端 WebSocket 错误处理缺失（Critical）

**位置**: `frontend/src/views/ReconstructionView.vue`

**问题**: WebSocket 连接失败时没有任何错误处理，用户无法知道连接是否成功

**影响**: 后端服务未启动或网络异常时，用户看不到任何错误提示，功能静默失效

---

### 1.7 前端内存泄漏（Critical）

**位置**: `frontend/src/views/DetectionView.vue`, `frontend/src/components/ReconstructionViewer.vue`

**问题**:
- 组件卸载时没有清理当前的 ObjectURL
- 每次调用 `updateMesh()` 都会创建新的光照对象，但旧的不会被释放

**影响**: 浏览器内存中残留未释放的 Blob 对象和光照对象

---

### 1.8 模型训练 VOC XML 解析缺少坐标校验（Critical）

**位置**: `model-training/yolo-training/dataset.py:186-194`

**问题**: 没有校验 `x1 >= x2` 或 `y1 >= y2` 的情况，会产生负宽度/高度

**影响**: 生成无效的 YOLO 标注，导致训练失败或结果异常

---

### 1.9 模型训练 seed 注释错误（High）

**位置**: `model-training/yolo-training/train.ipynb:121`

**问题**: `seed=0` 是固定种子，不是随机，但注释写着"0=随机"

**影响**: 误导用户，导致每次训练结果相同但用户以为是随机的

---

### 1.10 predict.py 结果与图片路径不匹配（High）

**位置**: `model-training/yolo-training/predict.py:63`

**问题**: `model.predict()` 返回结果的顺序与文件系统排序顺序不一定一致

**影响**: 检测结果与图片路径错误关联

---

### 1.11 coloring.py 畸变系数索引越界风险（High）

**位置**: `backend/reconstruction/coloring.py:91`

**问题**: 假设 `dist_coeff` 至少有5个元素，但如果属性存在但长度不足会崩溃

**影响**: 相机内参配置错误时会导致崩溃

---

### 1.12 scene_loader.py PLY 读取精度不一致（Medium）

**位置**: `backend/loader/scene_loader.py:168`

**问题**: 使用 Open3D 读取时返回 `float32`，但其他模块使用 `float64`

**影响**: 精度损失，可能导致坐标计算误差累积

---

### 1.13 前端 apiClient baseUrl 未初始化（Medium）

**位置**: `frontend/src/services/apiClient.ts:100`

**问题**: 如果用户未先调用 `setHost()` 就调用其他 API 方法，请求会发送到当前页面 URL

**影响**: 请求发送到错误地址，功能失效

### 1.14 train.ipynb 调用不存在的函数（Medium）

**位置**: `model-training/yolo-training/train.ipynb`

**问题**: `from predict import predict_image` - predict.py 中不存在此函数

### 1.15 lidar_camera_fusion 空深度数组导致崩溃（Medium）

**问题**: `color_map_distance()` 未处理空数组情况

### 1.16 lidar_camera_fusion backproject_pixel_to_3d 参数错误（Medium）

## 二、架构设计改进建议

### 2.1 后端全局状态管理重构

**问题**: `routes.py` 中直接实例化组件（`fusion`、`engine`、`_loader`、`_ws_clients`），无法进行测试隔离和配置管理

**建议**: 创建 `ApplicationState` 类统一管理所有全局状态，使用 FastAPI 的 `Depends` 进行依赖注入

---

### 2.2 配置管理分散

**问题**: 配置参数分散在 `main.py`、`engine.py`、`routes.py` 等多个文件中

**建议**: 使用 `pydantic-settings` 集中管理所有配置，支持配置文件 + 环境变量覆盖

---

### 2.3 lidar_camera_fusion 模块职责不清

**问题**: `lidar_camera_fusion.py` 包含了过多职责：独立运行模式、管线集成模式、工具函数

**建议重构**:
```
lidar_camera_fusion/
├── core/              # 核心算法
│   ├── projection.py  # 投影算法
│   ├── transformation.py  # 变换矩阵
│   └── filtering.py   # 点云过滤
├── io/                # 数据读写
├── visualization/     # 可视化
├── pipelines/         # 流水线
└── utils/             # 工具函数
```

---

### 2.4 lidar_camera_fusion 两个 main() 函数重复

**问题**: `main.py` 和 `lidar_camera_fusion.py` 都定义了 `main()` 函数，功能重复且容易混淆

**建议**: 保留 `main.py` 作为唯一入口，移除 `lidar_camera_fusion.py` 中的 `main()`

---

### 2.5 前端缺少全局状态管理

**问题**: `connected` 状态通过 props 逐层传递，难以维护

**建议**: 使用 **Pinia** 或 Vue 3 的 `provide/inject` 机制管理全局状态

---

### 2.6 前端代码重复严重

**问题**: `DetectionView.vue` 和 `CrackDetection.vue` 代码几乎完全相同，`CrackDetection.vue` 在路由中未被使用

**建议**: 删除 `CrackDetection.vue`，或提取公共逻辑为 composable

---

### 2.7 前端缺少 TypeScript 类型定义

**问题**: API 返回数据没有 TypeScript 类型定义，所有返回值都是 `any` 类型

**建议**: 创建 `src/types/api.ts` 定义接口

---

### 2.8 前端 API 客户端设计不完善

**问题**: 使用全局变量存储 `baseUrl`，线程不安全，没有统一的错误处理和超时机制

**建议**: 使用类封装，添加拦截器模式和超时机制

---

### 2.9 模型训练缺少统一配置管理

**问题**: 训练参数、模型路径、数据路径硬编码在 `train.ipynb` 和各脚本中

**建议**: 使用统一的配置文件（如 `config.yaml`）管理所有参数

---

### 2.10 模型训练训练逻辑与可视化混合

**问题**: `train.ipynb` 将训练逻辑、可视化、推理混合在一起

**建议**: 将训练逻辑提取到 `train.py` 脚本中

---

### 2.11 后端模块间解耦

**问题**: 模块之间直接导入，耦合度较高

**建议**: 定义清晰的接口层，使用抽象基类（ABC），通过依赖注入实现模块间的解耦

---

### 2.12 后端异步架构优化

**问题**: 所有 CPU 密集型操作通过 `run_in_executor` 提交到线程池，但使用方式不一致

**建议**: 使用 `asyncio.to_thread`（Python 3.9+）替代 `loop.run_in_executor(None, ...)`

---

## 三、过时技术和技术债

### 3.1 后端使用 threading.RLock 而非 asyncio.Lock

**问题**: 异步上下文中使用线程锁不合适

**建议**: 使用 `asyncio.Lock`

---

### 3.2 后端使用 gc.collect()

**问题**: 现代 Python 自动垃圾回收足够高效，手动调用不必要

**建议**: 移除

---

### 3.3 后端使用 os.getenv 分散调用

**问题**: 配置管理不规范

**建议**: 使用 `pydantic-settings`

---

### 3.4 前端 Vue Router 版本问题

**问题**: `vue-router: ^5.1.0` 不是 Vue 3 的标准版本，Vue 3 应该使用 `vue-router@4.x`

**建议**: 降级到 `vue-router@4.x`

---

### 3.5 前端 HTTP 请求使用原生 fetch

**问题**: 使用原生 `fetch` 缺少请求/响应拦截、统一错误处理、请求取消、请求重试

**建议**: 使用 **Axios** 或封装统一的 HTTP 客户端

---

### 3.6 模型训练依赖版本过旧

**位置**: `model-training/yolo-training/requirements.txt`

| 依赖 | 当前版本 | 推荐版本 |
|------|---------|---------|
| torch | ~=2.0.0 | ~=2.4.0 |
| ultralytics | ~=8.4.0 | ~=8.20.0 |
| numpy | ~=1.26.0 | ~=1.28.0 |

---

### 3.7 lidar_camera_fusion use_euler 配置命名不当

**问题**: `use_euler: false` 语义不明确

**建议**: 使用更明确的枚举值 `rotation_type: "matrix"` / `"quaternion"` / `"euler"`

---

### 3.8 使用 cv2.imread 读取图片

**问题**: `cv2.imread` 以 BGR 格式读取，与 Python 生态主流的 RGB 格式不一致

**建议**: 使用 `PIL.Image` 代替

---

### 3.9 coloring.py RGB/BGR 转换冗余

**问题**: 先将 RGB 转为 BGR，最后又转回 RGB，增加了不必要的转换开销

**建议**: 统一使用 RGB 格式

---

### 3.10 lidar_camera_fusion 硬编码的深度图例参数

**问题**: `_draw_distance_legend()` 中颜色条尺寸和位置是硬编码的

**建议**: 将这些参数放入配置文件的 `visualization` 部分

---

- `reconstruction/engine.py:45-46`: `from reconstruction.schemas import ...` → 应为 `from .schemas import ...`
