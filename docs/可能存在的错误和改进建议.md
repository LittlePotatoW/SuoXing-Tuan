## 一、严重 Bug 和错误

### 1.1 并发安全问题（Critical）

**位置**: `backend/reconstruction/routes.py`

```python
fusion = DataFusion(sensor_pose_in_body=LIDAR_POSE_IN_BODY)
engine = ReconstructionEngine()
_ws_clients: list[WebSocket] = []
```

**问题**:
- `ReconstructionEngine` 的 `add_frame()` 和 `_rebuild()` 方法没有线程安全保护，多个客户端同时上传帧会导致数据竞争
- `_ws_clients` 列表在遍历过程中被修改，可能导致 `RuntimeError: list changed size during iteration`

**影响**: 高并发场景下数据损坏、服务崩溃

---

### 1.2 InferenceEngine 锁使用错误（High）

**位置**: `backend/inference.py`

```python
def load(self, model_path: str):
    new_model = YOLO(model_path)  # 在锁外创建模型
    
    with self._lock:
        self._model = new_model  # 在锁内替换
```

**问题**: 模型创建在锁外执行，多个线程同时调用会创建多个模型实例，造成内存浪费和状态不一致

---

### 1.3 相机轨迹内存泄漏（High）

**位置**: `backend/reconstruction/engine.py`

```python
def add_frame(self, frame: FusedFrame) -> None:
    for cam in frame.cameras_world:
        self._camera_trail.append([cam.position.x, cam.position.y, cam.position.z])
```

**问题**: `_camera_trail` 列表只增不减，长时间运行后内存占用持续增长

---

### 1.4 predict.py 结果与图片路径不匹配（High）

**位置**: `model-training/yolo-training/predict.py`

```python
for r, img_path in zip(results, [str(Path(p)) for p in sorted(Path(source).glob("*"))]):
```

**问题**: `model.predict()` 返回结果的顺序与文件系统排序顺序不一定一致，导致检测结果与图片路径错误关联

---

### 1.5 WebSocket 资源泄漏（High）

**位置**: `frontend/src/views/ReconstructionView.vue`

**问题**:
- 创建新 WebSocket 时直接覆盖旧引用，未等待旧连接关闭
- 页面卸载时没有关闭 WebSocket 连接

---

### 1.6 Three.js 资源未释放（High）

**位置**: `frontend/src/components/ReconstructionViewer.vue`

```typescript
function updateMesh(data: ...) {
  meshGroup.clear()  // 只是从场景中移除，没有释放 GPU 资源
  // 创建新的 geo、mat、mesh...
}
```

**问题**: 每次更新都会创建新的 `BufferGeometry` 和 `Material`，但没有调用 `dispose()` 释放旧资源，导致 GPU 内存泄漏

---

### 1.7 apiClient baseUrl 未初始化（High）

**位置**: `frontend/src/services/apiClient.ts`

```typescript
let baseUrl = ''  // 初始为空
```

**问题**: 如果用户未先调用 `setHost()` 就调用其他 API 方法，请求会发送到当前页面 URL

---

### 1.8 train.ipynb 调用不存在的函数（High）

**位置**: `model-training/yolo-training/train.ipynb`

```python
from predict import predict_image  # 错误：predict.py 中不存在此函数
```

**问题**: 运行此 Cell 会抛出 `ImportError`

---

### 1.9 JSON 输出目录可能不存在（Medium）

**位置**: `model-training/yolo-training/predict.py`

### 1.10 模型加载无原子性保障（Medium）

**位置**: `backend/main.py`

**问题**: 如果写入过程中断，会留下不完整的模型文件，下次启动时自动加载会失败

### 1.11 缺少输入验证（Medium）

### 1.12 组件卸载后仍可能执行回调（Medium）

## 二、架构设计改进建议

### 2.1 后端缺少依赖注入和生命周期管理

**问题**: `routes.py` 中直接实例化组件，无法进行测试隔离和配置管理

**建议**: 使用 FastAPI 的 `Depends` 进行依赖注入

---

### 2.2 配置管理分散

**问题**: 配置参数分散在 `main.py`、`engine.py`、`routes.py` 等多个文件中

**建议**: 创建 `config.py` 使用 `pydantic_settings` 集中管理所有配置

---

### 2.3 缺少状态管理抽象

**问题**: `ReconstructionEngine` 混合了业务逻辑和状态管理

**建议**: 将状态管理抽取为独立类，使用策略模式支持不同的重建算法

---

### 2.4 WebSocket 管理不完善

**问题**: `ws_clients` 列表直接操作，没有封装管理类，缺少心跳检测和超时清理机制

**建议**: 创建 `WebSocketManager` 类统一管理

---

### 2.5 前端缺少全局状态管理

**问题**: `connected` 状态通过 props 逐层传递，难以维护

**建议**: 使用 **Pinia** 进行状态管理

---

### 2.6 前端组件职责不清晰

**问题**: `DragDropZone.vue` 同时负责文件上传和图片展示

**建议**: 将图片展示逻辑移至 `ImageViewer.vue`

---

### 2.7 前端缺少 Composable 复用

**问题**: `DetectionView.vue` 和 `CrackDetection.vue` 代码几乎完全相同，存在大量重复

**建议**: 提取 `useInference()` composable

---

### 2.8 模型训练缺少统一配置管理

**问题**: 所有脚本使用硬编码路径和参数

**建议**: 创建统一的配置文件（如 `config.yaml`）

---

### 2.9 模型训练缺少实验管理

**问题**: 每次训练的结果直接覆盖，无法保留历史实验记录

**建议**: 使用 `ultralytics` 的 `name` 参数自动生成带时间戳的实验名，或集成 MLflow

---

### 2.10 模型训练训练逻辑与可视化混合

**问题**: `train.ipynb` 将训练逻辑、可视化、推理混合在一起

**建议**: 将训练逻辑提取到 `train.py` 脚本中

---

## 三、过时技术和技术债

### 3.1 后端缺少异步优化

**问题**: `engine.add_frame()` 和 `engine._rebuild()` 是 CPU 密集型操作，在异步路由中直接调用会阻塞事件循环

**建议**: 将重建操作移到线程池中

---

### 3.2 全局常量使用不当

**位置**: `backend/reconstruction/engine.py`

```python
VOXEL_SIZE = 0.01
REBUILD_INTERVAL_FRAMES = 10
```

**问题**: 常量定义在模块级别，无法在运行时动态调整

**建议**: 使用 `dataclass` 或配置对象

---

### 3.3 缺少连接池和资源复用

**问题**: 每次推理都创建新的 `YOLO` 实例，没有模型缓存机制

**建议**: 实现模型缓存和连接池

---

### 3.4 缺少监控和指标

**问题**: 没有记录关键性能指标（推理时间、重建时间、内存使用）

**建议**: 添加 Prometheus 指标收集

---

### 3.5 前端未使用 Vue 3 组合式 API 最佳实践

**问题**: 没有使用 `composables` 组织代码

**建议**: 创建 `composables/` 目录，提取 `useApi.ts`、`useInference.ts`、`useWebSocket.ts`

---

### 3.6 前端缺少代码分割和懒加载

**问题**: 所有路由组件都在初始加载时同步导入

**建议**: 使用 Vue Router 的懒加载

---

### 3.7 前端 HTTP 请求使用原生 fetch

**问题**: 使用原生 `fetch` 缺少请求/响应拦截、统一错误处理、请求取消、请求重试

**建议**: 使用 **Axios** 或封装统一的 HTTP 客户端

---

### 3.8 前端缺少 TypeScript 类型定义

**问题**: 后端 API 返回的数据没有定义 TypeScript 接口

**建议**: 创建 `types/` 目录，定义检测结果、网格数据、WebSocket 消息等类型

---

### 3.9 模型训练依赖版本过旧

**位置**: `model-training/yolo-training/requirements.txt`

| 依赖 | 当前版本 | 推荐版本 |
|------|---------|---------|
| torch | ~=2.0.0 | ~=2.4.0 |
| ultralytics | ~=8.4.0 | ~=8.20.0 |
| numpy | ~=1.26.0 | ~=1.28.0 |

---

### 3.10 使用 cv2.imread 读取图片

**问题**: `cv2.imread` 以 BGR 格式读取，与 Python 生态主流的 RGB 格式不一致

**建议**: 使用 `PIL.Image` 代替

---
