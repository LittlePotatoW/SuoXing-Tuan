# 可能存在的错误和改进建议

> 基于对全部代码的审查，涵盖后端（Python FastAPI）、前端（Vue 3 + TypeScript + Three.js）、算法层、配置、测试。
> 每个 bug 附带修复后的潜在副作用。

## 一、严重 Bug（P0 — 影响正确性/安全性/稳定性）

### 1.1 路径遍历漏洞：Session / Report 端点可读写任意文件

**文件:** [session.py](..\backend\server\api\routes\session.py):25, 58; [report.py](..\backend\server\api\routes\report.py):22, 41
**类型:** [BUG] 安全

`body.name`、`session_name`、`file_path`、`body.filename` 均未拒绝 `../` 等路径遍历字符。攻击者可读写 `Session_Data/` / `Report_Data/` 目录外的任意文件。

**修复:** 使用 `Path.resolve()` 并验证结果前缀为对应数据目录。对文件名加正则约束 `^[\w\-\.]+$`。
**副作用:** 已有带特殊字符的 session/report 文件将无法访问，需迁移。

---

### 1.2 EKF 融合：视觉里程计增量未从机体坐标转换到全局坐标

**文件:** [fusion.py](..\backend\server\estimation\fusion.py):68
**类型:** [BUG] 算法正确性

`z = np.array([x + dx, y + dz, heading + dh])` — `dx`（右向）和 `dz`（前向）是机体坐标系，直接加到全局 `(x, y)` 上，未按航向角旋转。

**修复:** 先用航向角旋转 `(dx, dz)` 得到全局增量，再加到全局坐标。
**副作用:** EKF 融合轨迹会发生显著变化，需重新验证所有依赖位置估计的下游逻辑。

---

### 1.3 RGB-D 视觉里程计：航向角符号取反

**文件:** [rgbd_odometry.py](..\backend\server\estimation\rgbd_odometry.py):90
**类型:** [BUG] 算法正确性

`dheading = math.degrees(math.atan2(T[2, 0], T[2, 2]))` — 从旋转矩阵 `R[2,0] = -sin(yaw)` 推导，`atan2(-sin, cos) = -yaw`，符号取反。左转会记录为右转。

**修复:** `dheading = -math.degrees(math.atan2(T[2, 0], T[2, 2]))`
**副作用:** 轨迹转弯方向会反转，已有录制数据的重建结果将完全不同。

---

### 1.4 视觉里程计增量旋转中 `dx`/`dz` 交换

**文件:** [facade.py](..\backend\server\estimation\facade.py):140-141
**类型:** [BUG] 算法正确性

OD 模式中将 `dx`（相机右）乘 cos、`dz`（相机前）乘 sin，但前向分量应匹配 cos 项。

**修复:** 交换 `dx`/`dz` 对应的三角函数项。
**副作用:** 轨迹方向偏转修正，已有位置数据不再对齐。

---

### 1.5 `map_to_3d` 与系统点云表示不兼容

**文件:** [postprocess.py](..\backend\server\detection\postprocess.py):84
**类型:** [BUG] 运行时崩溃

`map_to_3d` 期望 `(H, W, 3)` 有序点云，并使用 2D 切片索引。但 `depth_to_pointcloud` 返回 `(N, 3)` 无序点云。调用时传入 `None` 作为深度图（[engine.py](..\backend\server\engine\engine.py):240），直接 `AttributeError`。

**修复:** 要么让 `depth_to_pointcloud` 同时输出有序点云，要么重写 `map_to_3d` 用 KD 树做空间裁剪查询。
**副作用:** 需重构点云数据流，改动面较大。

---

### 1.6 NMS 使用 `intersection/area_other` 而非标准 IoU

**文件:** [postprocess.py](..\backend\server\detection\postprocess.py):57
**类型:** [BUG] 算法正确性

`overlap = (w * h) / areas[order[1:]]` 是 `intersection / area_other`，非标准 IoU（应为 `intersection / union`）。大框包含小框时比值=1.0，小框与大框比较时比值很小，抑制行为不对称。

**修复:** 改为 `overlap = (w * h) / (areas[i] + areas[order[1:]] - (w * h))`
**副作用:** NMS 抑制结果发生变化，保留的检测框集合不同。

---

### 1.7 后端 Session save：内存崩溃 + 单次请求发送全部帧数据

**文件:** [data-saver.ts](..\frontend\src\services\data-saver\data-saver.ts):30-36
**类型:** [BUG] 稳定性

`recordFrame` 将所有帧的 base64 图像全部累积在内存数组中，`finalize()` 一次性 POST。几百帧即可耗尽浏览器内存，且单个 HTTP 请求体远超服务器默认限制。

**修复:** 分批上传（每 N 帧一个批次），或将帧写入 IndexedDB 后流式发送。
**副作用:** 需修改前后端保存协议，增加部分上传失败的恢复逻辑。

---

### 1.8 `_save_pointcloud` 异常发生在 `flush()` 之后 → 数据丢失

**文件:** [engine.py](..\backend\server\engine\engine.py):257
**类型:** [BUG] 数据安全

`_trigger()` 在 `flush()` 之后才执行 `_save_pointcloud()`。若文件写入阶段抛异常（磁盘满/权限），帧缓冲已被清空，数据即丢失。

**修复:** 将 flush 之后的操作全部包在 try/except 中，或先保存再 flush。
**副作用:** 异常时可能产生重复数据，需引入去重机制。

---

### 1.9 增量模式点云无限膨胀 → OOM

**文件:** [engine.py](..\backend\server\engine\engine.py):213-214
**类型:** [BUG] 内存

增量模式下 `_latest_result.point_cloud` 持续累积所有帧的点，且 `surface.enabled` 时 `skip_downsample=True`，点云永不降采样。100 轮重建后可达数百万点，Poisson 重建耗时分钟级甚至 OOM。

**修复:** 引入滑动窗口或对存量点云始终降采样。
**副作用:** 旧数据会被淡忘，需告知用户新的重建语义。

---

### 1.10 前端 PLY 颜色偏移计算在非标准布局中错误

**文件:** [point-cloud.ts](..\frontend\src\services\renderer\point-cloud.ts):129
**类型:** [BUG] 运行时渲染错误

`colorOffset` 被硬编码计算为 `vertStride - (hasColor ? 3 : 0)`，假设颜色始终在顶点结构末尾。若颜色在中间（如 `x y z r g b nx ny nz`），颜色读取为乱码。

**修复:** 解析头时记录第一个颜色属性的精确字节偏移。
**副作用:** 对各种合法 PLY 布局更鲁棒。

---

### 1.11 PLY 面索引使用 `getInt32` 读取无符号整数

**文件:** [point-cloud.ts](..\frontend\src\services\renderer\point-cloud.ts):149

**修复:** 改用 `getUint32`。
**副作用:** 无（纯正确性修复）。

## 二、设计问题（架构 / 模式 / 耦合）

### 2.1 EKF 雅可比矩阵简化为单位阵，协方差传播不完整

**文件:** [fusion.py](..\backend\server\estimation\fusion.py):53
**类型:** [DESIGN]

`F = np.eye(STATE_DIM)` 忽略了航向角-位置之间的非线性耦合。`P` 只在过程噪声项增长，缺失交叉耦合导致位置不确定性被低估，卡尔曼增益不准确。

**修复:** 计算真实雅可比矩阵 `F = [[1, 0, -v*dt*sin(heading)], [0, 1, v*dt*cos(heading)], [0, 0, 1]]`。
**副作用:** EKF 收敛行为改变，需重新调 Q/R 噪声矩阵。

---

### 2.2 同步重建阻塞 HTTP 工作线程

**文件:** [engine.py](..\backend\server\engine\engine.py):166-278
**类型:** [DESIGN] 架构

`_trigger()` 在线程池工作线程上同步执行 5-30 秒的计算（基 64 解码、深度转点云、坐标变换、Open3D 降采样、YOLO 推理、Poisson 重建、文件 I/O）。5 个并发重建即可耗尽 40 线程的默认线程池，所有 API 请求被饿死。

**修复:** 将重建卸载到专用后台线程或使用 `asyncio.to_thread` + 任务队列。`push_frame` 立即返回。
**副作用:** 大规模架构变更，但前端轮询模式已兼容异步结果。

---

### 2.3 三个独立全局单例生命周期无协调

**文件:** [engine.py](..\backend\server\engine\engine.py); [facade.py](..\backend\server\estimation\facade.py); [detector.py](..\backend\server\detection\detector.py)
**类型:** [DESIGN]

`ReconstructionEngine`、`PositionEstimator`、`Detector` 各有独立的 `_instance`/`_lock`/`create()`/`stop()`，无协调机制。停掉估计器后引擎仍持有旧位置引用。

**修复:** 引入统一的 ServiceRegistry 或依赖注入管理生命周期。
**副作用:** 重构面较大。

---

### 2.4 静态文件挂载路径 vs 保存路径不一致

**文件:** [main.py](..\backend\server\main.py):55-57; [engine.py](..\backend\server\engine\engine.py):283
**类型:** [DESIGN]

`create_app()` 按 `__file__` 绝对路径挂载 `output/`，但 `_save_pointcloud()` 按 `os.getcwd()` 相对路径保存。若 CWD 不是 `backend/`，保存的文件 404。

**修复:** 统一使用基于 `__file__` 的绝对路径。
**副作用:** 若用户依赖相对路径行为，输出文件位置会移动。

---

### 2.5 配置加载依赖 CWD，非 `__file__` 锚定

**修复:** 从 `__file__`（本模块所在目录）开始搜索。
**副作用:** 无。

### 2.6 `get_config()` 竞态：双重检查锁缺少锁

### 2.7 自行车模型使用一阶欧拉积分

**文件:** [bicycle.py](..\backend\server\estimation\bicycle.py):31-35
**类型:** [DESIGN]

转弯时用初速航向角积分而非解析圆弧。`delta_t` 较大时误差显著。
