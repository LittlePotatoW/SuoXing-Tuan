# 实时融合页面设计方案

## 1. 模式总览

页面保留**被动**和**主动**两种顶层模式。主动模式扩展为三种数据源子模式：

```
被动 (Passive)
  └─ 不主动拉取数据，仅通过 WebSocket 接收后端重建结果

主动 (Active)
  ├─ 中继 (Relay)   — 轮询 TranspondServer，实时拉取数据
  ├─ 回放 (Replay)  — 从 TranspondServer 按范围查询历史数据并播放
  └─ 手动 (Manual)  — 从本地文件夹加载已保存的数据
```

---

## 2. 主动模式 — 三种数据源

### 2.1 中继 Relay（与现有逻辑一致）

- 输入 TranspondServer 地址
- 选择轮询间隔（1s / 2s / 5s）
- 启动后循环轮询 `/location` 和 `/sensor`，转发给后端

### 2.2 回放 Replay（从 TranspondServer 拉取历史数据）

两种范围指定方式，二选一：

**方式 A — 按帧数量**：
| 参数 | 说明 | 示例 |
|------|------|------|
| 从最近第 N 份 | 从倒数第 N 条开始 | 最近 50 份 |
| 到最近第 M 份（可选，默认0=当前） | 到倒数第 M 条结束 | 到最近 10 份 |

即拉取 `[latest-N, latest-M]` 范围的数据（时间升序播放）。

**方式 B — 按时间范围**：
| 参数 | 说明 | 示例 |
|------|------|------|
| 开始时间 | 多久以前 | 5 分钟前 |
| 结束时间（可选，默认=现在） | 多久以前 | 1 分钟前 |

即拉取 `[now-start, now-end]` 范围的数据。

**回放控制**：
- [加载] 按钮 — 先按范围 GET TranspondServer 拉取数据到浏览器内存
- 显示拉取结果：loc=N 条, sensor=M 条
- [播放] / [暂停] / [停止]
- 速度调节：0.5x / 1x / 2x / 5x
- 进度条 + 当前帧号显示

**数据转发**：回放时按原始时间戳间隔依次将 loc 和 sensor 数据 POST 给后端处理，与中继模式使用相同的转发逻辑。

### 2.3 手动 Manual（从本地文件夹加载）

**文件夹结构**（位于项目根目录 `data_captures/`，已加入 `.gitignore`）：

```
data_captures/
  <session_name>/
    location/         ← 定位数据，每帧一个 JSON 文件
      00001.json
      00002.json
      ...
    sensor/           ← 传感器数据，每帧一个 JSON 文件
      00001.json
      00002.json
      ...
    fusion/           ← 融合重建结果，每次重建一个 JSON 文件
      00001.json
      00002.json
      ...
```

**手动模式加载流程**：

1. 用户输入文件夹名（session name）
2. 点击 [加载]，前端读取 `data_captures/<name>/` 目录
3. **数据完整性判断**：

| 情况 | loc/ | sensor/ | fusion/ | 行为 |
|------|------|---------|---------|------|
| A | 有数据 | 有数据 | 可有可无 | 将 loc+sensor 逐帧发送给后端融合→渲染 |
| B | 缺失或不完整 | 缺失或不完整 | 有数据 | **跳过融合**，直接用 fusion 数据渲染 |
| C | 其他组合 | — | — | 提示"数据缺失：需要 (loc+sensor) 或 fusion 数据" |

- 情况 A：按时间戳对齐 loc 和 sensor，按序 POST 给后端处理，通过 WebSocket 接收重建结果
- 情况 B：fusion JSON 中应包含 mesh 数据（顶点/面/颜色）、缺陷列表、轨迹等，直接调用 `viewerRef.addMesh()` 等方法渲染

## 3. 保存模式（任意模式可开启）

在**中继、回放、手动**任意模式下，可独立开启保存模式。

**控件**：
- 保存开关（checkbox）
- 保存名称输入框
- 数据类型勾选：`[x] Loc  [x] Det  [x] Fusion`

**行为**：
- 开启后，所有通过管线的数据同步写入 `data_captures/<name>/` 对应子目录
- Loc 数据：每收到/生成一帧 → 写入 `location/00001.json`, `00002.json` ...
- Det 数据：每收到/生成一帧 → 写入 `sensor/00001.json`, `00002.json` ...
- Fusion 数据：每次后端 WebSocket 推送 `rebuild_complete` → 写入 `fusion/00001.json`, `00002.json` ...
- 关闭保存开关后停止写入

**适用场景**：
- 中继 + 保存：录制真实小车数据
- 回放 + 保存：从 TranspondServer 导出历史数据到本地
- 手动 + 保存：回放已有数据时同时保存融合结果

## 4. 匀速直线模式（任意模式可用）

在**中继、回放、手动**任意模式下，可开启匀速直线模式。

**控件**：
- 匀速直线开关（checkbox）
- 速度输入框（m/s，默认 0.5）

**行为**：
- 开启后，**忽略所有 loc 数据**（不从 TranspondServer 拉取、不发送给后端）
- 每条 sensor/det 帧发送给后端时，其 `car_position` 替换为匀速直线推算：
  - `x = speed * (timestamp_ns - t0) / 1e9`
  - `y = 0, z = 0, yaw = 0`
- `t0` = 第一帧 sensor 的 `timestamp_ns`

**适用场景**：
- 小车定位模块故障时，单独调试检测端
- 不需要真实轨迹，只想看传感器数据的重建效果

- **中继模式**显示：TranspondServer 地址、轮询间隔、连接状态

## 8. 数据格式约定

### 保存的 location 帧 (`data_captures/<name>/location/XXXXX.json`)

直接保存 TranspondServer 返回的原始 location 数据：
```json
{
  "timestamp_ns": 1783581801179000000,
  "camera": [{ "camera_pose": { "position": {...}, "rotation": {...} } }],
  "car": { "kinematics": { "velocity": 0.5, "steering_angle": 0.0, "wheel_base": 1.5 } }
}
```

### 保存的 sensor 帧 (`data_captures/<name>/sensor/XXXXX.json`)

直接保存 TranspondServer 返回的原始 sensor 数据（单个 frame，不含 batch 外层）：
```json
{
  "frame_id": "f000039",
  "timestamp_ns": 1783586485847000000,
  "point_cloud": { "points": [...], "point_count": 300 },
  "camera_views": [{ "image_data": "...", "width": 544, "height": 384, ... }],
  "car_position": { "pose": {...}, "timestamp_ns": ... },
  "kinematics": { "velocity": 0.5, ... }
}
```

### 保存的 fusion 数据 (`data_captures/<name>/fusion/XXXXX.json`)

保存后端 WebSocket `rebuild_complete` 消息中的 `data` 字段：
```json
{
  "mesh": { "vertices": [...], "faces": [...], "colors": [...] },
  "camera_trail": [{ "x": 0, "y": 0, "z": 0 }, ...],
  "cracks": [{ "position": {...}, "crack_type": "裂缝", "confidence": 0.95 }, ...],
  "total_frames": 15
}
```
